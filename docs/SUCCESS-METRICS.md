# Success Metrics

How we measure whether agentOW is working. Optimized for **quality** (do the PRs hold up?) and **reporting upward** (credible, comparable, defensible numbers).

> **Status:** measurement framework defined; data collection in progress. Numbers in the tables below are placeholders (`—`) until a statistically meaningful batch (~20–30 PRs) is collected.

---

## North Star: Merge Rate (without major rework)

> Of all PRs generated by agentOW, the share that a human reviewer approves and merges, **with less than {REWORK_THRESHOLD}% of the lines changed by humans before merge**.

One number that answers "is the quality good enough?" The "without major rework" qualifier is essential — a PR that gets merged but rewritten by the reviewer is not a success, and counting it as one is self-deception.

**Definition of "major rework":**
```
rework ratio = (lines changed by humans on top of agentOW's commits, before merge)
             ÷ (total lines agentOW changed)
```
A PR counts toward the North Star only if `rework ratio < {REWORK_THRESHOLD}%`.

> ⚠️ **Assumption — needs team sign-off:** `REWORK_THRESHOLD = 20%`. This is a placeholder. Stricter review cultures may want 10%; looser ones 30%. Align with the team before reporting.

## Supporting Metrics (explain the North Star)

| Metric | Definition | Reporting value |
|--------|------------|-----------------|
| **First-pass rate** | Share of PRs merged with **zero** reviewer must-fix comments | Proves quality isn't propped up by human cleanup |
| **Reviewer rework** | Avg lines changed / must-fix comments per merged PR | Quantifies human effort saved vs. created |
| **Time-savings multiple** | (human time for a comparable task) ÷ (agentOW time to draft PR) | The "Nx efficiency" line reports love |

| Metric | Value |
|--------|-------|
| First-pass rate | —% |
| Avg reviewer rework (lines / comments) | — / — |
| Time-savings multiple | —x |

> ⚠️ **Assumption — needs a source:** the numerator for time-savings ("human time for a comparable task") is the most-questioned number in any report. Decide the source before quoting it: (a) historical ADO cycle-time data, (b) a one-time timed comparison on a sample of tasks, or (c) explicitly labeled estimates. Do NOT report an unsourced multiple.

## Guardrail Metrics (prevent gaming the North Star)

The two questions a sharp reviewer will ask: "Did you only pick easy tasks?" and "Did the merged PRs introduce bugs?" These guardrails answer them preemptively.

| Guardrail | Definition | Prevents |
|-----------|------------|----------|
| **Complexity distribution** | Tasks bucketed by difficulty, with per-bucket merge rate | "Only attempting trivial tasks to inflate the score" |
| **Regression rate** | Bugs introduced / reverts within N weeks of merge | "Fast but plants landmines" |

**Complexity buckets** (assign each task one bucket at intake):

| Bucket | Definition | PRs | Merge rate |
|--------|------------|-----|------------|
| Trivial | typo, string change, single-line | — | —% |
| Single-component | one component, no cross-file coordination | — | —% |
| Cross-file | multiple files, coordinated change | — | —% |
| Gated | involves killswitch / flight / feature gate | — | —% |

**Regression:**

| Metric | Value |
|--------|-------|
| Reverted within N weeks | —% |
| Bugs traced to an agentOW PR | — |

> ⚠️ **Assumption — needs a mechanism:** regression rate requires linking "this bug came from that PR." If no such linkage exists, downgrade this guardrail to **revert rate** (objectively trackable from git) and note the gap. Set `N` (the observation window) to a value that matches your release cadence — placeholder `N = 4 weeks`.

## Data Collection

**Build measurement before quoting numbers.** With few users today, now is the time to instrument.

1. **Tag every agentOW PR.** The PR description already carries a `🤖`-style marker; ensure it's machine-detectable (a stable string or a label) so PRs can be filtered from ADO.
2. **At intake, record the complexity bucket** for each task (the planner or orchestrator can write it into `report.json`).
3. **Pull from ADO + git** (all objective, reproducible — no subjective scoring):
   - PR state (merged / abandoned / active)
   - diff between agentOW's last commit and the merge commit → rework ratio
   - reviewer comment threads → must-fix count
   - linked work items / reverts → regression
4. **Wait for ~20–30 merged PRs** before computing rates. Smaller samples are not defensible in a report.

---

## Why this design

- **One North Star** — a single sentence in a report, no audience getting lost in ten numbers.
- **Supporting metrics decompose it** — ready answers when asked "why is the rate that number?"
- **Guardrails prevent self-deception** — a 95% merge rate with no complexity/regression context is worthless to a smart reviewer.
- **Everything is objectively sourced from ADO + git** — reproducible and auditable, not vibes.

---

## What this is NOT

Internal loop-health metrics (convergence cycles, stall rate, token cost, screenshot-validation hit rate) are **diagnostics**, not success criteria. They explain *why* quality or adoption is what it is — they don't prove the harness is valuable. Track them in dashboards for debugging; keep them out of the success scorecard.
